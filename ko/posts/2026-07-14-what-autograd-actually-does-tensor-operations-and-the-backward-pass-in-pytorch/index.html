autograd가 실제로 하는 일: PyTorch 텐서 연산과 역전파의 내부

데이터과학
인공지능
PyTorch
Autograd
역전파
텐서

PyTorch의 .backward()는 대개 정답 그래디언트를 돌려주는 블랙박스로 다뤄집니다. 이 글은 그 상자를 엽니다. 리프 텐서, 텐서 연산이 만드는 동적 그래프, 역전파가 연쇄 법칙을 실행하는 방식을 해석 미분·수치 미분으로 검증하고, nn.Module과 옵티마이저 없이 autograd만으로 소규모 신경망을 학습시켜 그 구조를 눈으로 확인합니다.

한 줄 아래에 감춰진 것

모든 PyTorch 학습 루프에는 같은 세 줄이 들어 있습니다.

loss = criterion(model(x), y)
loss.backward()
optimizer.step()

가운데 한 줄이 딥러닝을 가능하게 만드는 계산을 담당합니다. 손실을 모든 파라미터에 대해 미분한 그래디언트를 계산하는 일입니다. 이 한 줄이, 예전에는 손으로 몇 페이지에 걸쳐 유도하던 도함수를 대체하며, 그것도 순전파 동안 PyTorch가 조용히 쌓아 둔 그래프를 되짚어 계산해 냅니다. 표준 모듈만 쓰는 연구 코드에서는 backward()를 블랙박스로 다뤄도 문제가 없습니다. 하지만 커스텀 손실, 비표준 층, 물리 기반 규제항, 여러 목표를 섞은 학습 루프를 직접 짜는 순간, 우리가 상상한 그래프와 PyTorch가 실제로 만든 그래프가 달라지면서 그래디언트가 조용히 틀려집니다.

이 글이 다루려는 것은 대부분의 튜토리얼이 건너뛰는 층입니다. requires_grad가 정확히 무엇을 표시하는지, 텐서 연산이 어떻게 계산 그래프를 조립하는지, backward()가 단계별로 무엇을 하는지입니다. 그 동작을 해석 미분(폐쇄형으로 정답이 알려진 경우)과 수치 미분 (autograd를 검증 대상 오라클로 두는 유한 차분 확인)으로 각각 검증한 뒤, 마지막에는 nn.Module과 torch.optim 없이 autograd만으로 소규모 신경망을 학습시켜 루프의 내부를 투명하게 드러냅니다. 시스템 설계와 아래에서 쓰는 용어는 Paszke 기타 (2019년) 를, 그 바탕이 되는 역방향 자동미분은 Baydin 기타 (2018년) 과 정형화된 서술인 Griewank 와/과 Walther (2008년) 을 따릅니다.

리프 텐서, requires_grad, 그리고 그래프 노드의 정체

PyTorch 텐서는 단순한 수치 배열이 아닙니다. autograd에서 중요한 세 가지 포인터를 함께 갖습니다. requires_grad(불리언), grad_fn(이 텐서를 만든 연산; 사용자가 직접 만든 경우에는 None), grad(backward()가 채우는 누적 그래디언트 버퍼)입니다. grad_fn이 None이면서 requires_grad가 True인 텐서를 리프(leaf) 텐서라 부릅니다. 모델의 파라미터는 리프 텐서이고, 그로부터 계산된 결과는 리프가 아닙니다.

import torch

torch.manual_seed(0)

x = torch.tensor([2.0, 3.0], requires_grad=True)  # 리프
y = (x ** 2).sum() + 3 * x[0] * x[1]              # 리프가 아님

print("x.requires_grad:", x.requires_grad, "| x.is_leaf:", x.is_leaf,
      "| x.grad_fn:", x.grad_fn)
print("y.requires_grad:", y.requires_grad, "| y.is_leaf:", y.is_leaf,
      "| y.grad_fn:", y.grad_fn)
x.requires_grad: True | x.is_leaf: True | x.grad_fn: None
y.requires_grad: True | y.is_leaf: False | y.grad_fn: <AddBackward0 object at 0x106a19750>

y의 순전파는 단순히 값 하나를 계산한 게 아닙니다. 연산자(**, sum, *, +) 하나 하나가 방향 있는 비순환 그래프에 노드를 붙였고, 각 노드는 역방향에서 지역 야코비언을 계산하는 데 필요한 정보를 저장합니다. 이 그래프는 동적입니다. 순전파를 할 때마다 새로 만들어지고 backward()가 끝나면 별도 지시가 없는 한 파기됩니다. y.backward()를 호출하면 PyTorch는 이 그래프를 역방향 위상 정렬 순서로 훑으며 지역 야코비언을 곱해 갑니다. 이것이 연쇄 법칙을, 간선마다 한 번씩 실행하는 방식입니다.

\(y = x_0^2 + x_1^2 + 3 x_0 x_1\) 의 도함수는

\[ \frac{\partial y}{\partial x_0} = 2 x_0 + 3 x_1, \qquad \frac{\partial y}{\partial x_1} = 2 x_1 + 3 x_0 \tag{1}\]

이므로 \(x = (2, 3)\) 에서 그래디언트는 \((2\cdot 2 + 3\cdot 3,\ 2\cdot 3 + 3\cdot 2) = (13, 12)\) 이어야 합니다.

y.backward()
print("y       =", y.item())
print("x.grad  =", x.grad.tolist())
print("analytic:", [2*2.0 + 3*3.0, 2*3.0 + 3*2.0])
y       = 31.0
x.grad  = [13.0, 12.0]
analytic: [13.0, 12.0]

이 출력에서 짚어야 할 두 가지가 있습니다. 첫째, x.grad는 호출 전에는 None이었고 호출 후에 값이 채워집니다. 이때 값은 누적되지 덮어씌워지지 않으므로 학습 루프 맨 위에서 zero_grad()를 부르는 것입니다. 둘째, 인자 없이 backward()를 부르려면 y가 스칼라여야 합니다. 스칼라가 아닌 출력에는 상류 그래디언트를 지정하는 grad_outputs가 필요합니다. 초심자가 배치 손실을 축소하지 않고 backward()를 호출했을 때 만나는 “grad can be implicitly created only for scalar outputs” 오류가 이 조건 때문에 생깁니다.

역방향 연쇄 법칙: 역전파의 골격

순전파가 그래프를 만든다면, 역전파는 그것을 되짚어 갑니다. 스칼라 손실 \(L\) 과 연산 \(z = f(a, b)\) 로 만들어진 중간 텐서 \(z\) 에 대해 지역 규칙은

\[ \frac{\partial L}{\partial a} = \frac{\partial L}{\partial z} \cdot \frac{\partial z}{\partial a}, \qquad \frac{\partial L}{\partial b} = \frac{\partial L}{\partial z} \cdot \frac{\partial z}{\partial b} \tag{2}\]

로 쓸 수 있습니다. Griewank 와/과 Walther (2008년) 이 정형화한 역방향 자동미분은, 정확히 이 규칙을 역방향 위상 정렬 순서로 노드마다 한 번씩 적용하는 것과 같습니다. 그 결과 리프 텐서에 도달할 즈음이면 \(\partial L / \partial \text{leaf}\) 가 그 리프에서 \(L\) 로 이어지는 모든 경로에 대한 합으로 축적되어 있습니다. 소스가 리프고 싱크가 스칼라 손실인 DAG라는 그래프 구조 덕분에 이 계산이 효율적입니다. 중간 결과 \(\partial L / \partial z\) 를 \(z\) 를 지나는 모든 경로가 공유하므로, 역방향 모드는 파라미터 수와 무관하게 한 번의 순전파 정도의 추가 비용으로 끝납니다 (Baydin 기타 2018년).

그림 1 는 위 소식의 그래프를 그린 것입니다. 실선은 순전파에서 만들어진 데이터 의존성이고, 점선 화살표는 y.backward() 를 실행할 때 각 간선을 따라 뒤로 전파되는 지역 편미분을 나타냅니다.

그림은 장식이 아닙니다. x0.grad가 두 갈래에서 들어오는 기여의 합이라는 사실을 보여 줍니다. 한쪽은 \(x_0^2 \to y\) 로, 다른 한쪽은 \(x_0 x_1 \to 3 x_0 x_1 \to y\) 로 도착합니다. 방정식 1 의 \(2 x_0\) 과 \(3 x_1\) 이 바로 여기서 나옵니다. 리프 텐서가 표현식에 두 번 이상 나타나면 그래디언트에도 그만큼 여러 번 나타난다는 규칙이 이 이유입니다.

온전성 검사: autograd 대 수치 미분

위의 장난감 예제보다 조금이라도 복잡해지면 손으로 해석 미분을 쓰는 것이 불가능해집니다. 그것이 autograd를 쓰는 이유이기도 합니다. 하지만 여전히 검증은 가능합니다. 유한 차분 그래디언트

\[ \frac{\partial L}{\partial \theta_i} \;\approx\; \frac{L(\theta + \varepsilon e_i) - L(\theta - \varepsilon e_i)}{2 \varepsilon} \tag{3}\]

는 절단 오차 \(\mathcal{O}(\varepsilon^2)\) 에 작동 정밀도 수준의 반올림 오차를 더한 것뿐이 어서, autograd를 대조할 블랙박스 기준으로 쓸 수 있습니다. 새 커스텀 연산자가 이상하게 굴 때 실무자가 습관처럼 짜는 검사가 바로 이것입니다. autograd와 유한 차분이 배정도 잡음 수준까지 일치하면 연산자는 거의 확실히 옳고, 일치하지 않으면 둘 중 하나가 틀린 것이고 보통은 새 코드 쪽입니다.

def f(w, x):
    W1, W2 = w
    h = torch.tanh(x @ W1)
    return (h @ W2).sum()

x  = torch.randn(4, 3, dtype=torch.float64)
W1 = torch.randn(3, 5, dtype=torch.float64, requires_grad=True)
W2 = torch.randn(5,    dtype=torch.float64, requires_grad=True)
w  = [W1, W2]

y = f(w, x)
y.backward()

def numerical_grad(f, w, x, eps=1e-4):
    grads = []
    for p in w:
        g = torch.zeros_like(p)
        flat_g = g.view(-1); flat_p = p.data.view(-1)
        for i in range(flat_p.numel()):
            orig = flat_p[i].item()
            flat_p[i] = orig + eps; y_pos = f(w, x).item()
            flat_p[i] = orig - eps; y_neg = f(w, x).item()
            flat_p[i] = orig
            flat_g[i] = (y_pos - y_neg) / (2 * eps)
        grads.append(g)
    return grads

num = numerical_grad(f, w, x)
print(f"max |autograd - numeric| for W1: {(W1.grad - num[0]).abs().max():.3e}")
print(f"max |autograd - numeric| for W2: {(W2.grad - num[1]).abs().max():.3e}")
max |autograd - numeric| for W1: 6.617e-09
max |autograd - numeric| for W2: 1.187e-12

두 값 모두 유한 정밀도의 잡음 수준입니다. 실제로 쓰는 그래디언트 성분보다 몇 자릿수 아래이므로, 통과한 그래디언트 검사에서 기대하는 결과입니다. 같은 절차가 torch.autograd.gradcheck 내부에서 도는 절차이며, 커스텀 torch.autograd.Function을 작성하면서 gradcheck 단계를 생략하는 것은 조용한 버그를 학습 파이프라인에 밀어 넣는 잘 알려진 방법입니다.

autograd만으로 짠 학습 루프

그래프와 역전파를 손에 잡히도록 만들기 위해, nn.Module도 torch.optim도 쓰지 않는 학습 루프를 전부 손으로 써 봅니다. \(\tanh\) 은닉층을 하나 갖는 소규모 신경망으로 \(y = 2\sin(x) + 0.3 x\) 에 잡음을 얹은 데이터를 학습합니다. 파라미터 갱신은 반드시 torch.no_grad() 블록 안에서 수행합니다. 갱신 자체가 그래프의 일부가 되어서는 안 되기 때문입니다.

torch.manual_seed(1)
N = 200
x_data = torch.linspace(-3, 3, N).unsqueeze(1)
y_true = 2 * torch.sin(x_data) + 0.3 * x_data
y_data = y_true + 0.1 * torch.randn_like(y_true)

D_in, D_h, D_out = 1, 16, 1
W1 = (torch.randn(D_in, D_h) * 0.5).requires_grad_(True)
b1 = torch.zeros(D_h,           requires_grad=True)
W2 = (torch.randn(D_h, D_out) * 0.5).requires_grad_(True)
b2 = torch.zeros(D_out,         requires_grad=True)
params = [W1, b1, W2, b2]

lr, losses = 5e-2, []
for step in range(600):
    h      = torch.tanh(x_data @ W1 + b1)
    y_pred = h @ W2 + b2
    loss   = ((y_pred - y_data) ** 2).mean()
    loss.backward()
    with torch.no_grad():
        for p in params:
            p -= lr * p.grad
            p.grad.zero_()
    losses.append(loss.item())

print(f"initial loss: {losses[0]:.4f}")
print(f"final   loss: {losses[-1]:.4f}")
initial loss: 19.4342
final   loss: 0.0513

이 루프에서 눈여겨볼 지점이 세 곳 있습니다. 먼저 with torch.no_grad() 블록입니다. p -= lr * p.grad 자체가 텐서 연산이라, 이 블록 밖에서 실행하면 다음 반복까지 그래프가 확장되어 결국 메모리가 폭발합니다. 다음은 p.grad.zero_() 입니다. 그래디언트는 여러 backward() 호출에 걸쳐 누적되므로, 초기화를 잊으면 매 스텝의 그래디언트가 이전 스텝의 그래디언트로 오염됩니다. 마지막은, 이 루프가 매 반복마다 그래프를 처음부터 다시 만든다는 사실입니다. PyTorch의 이른바 “define-by-run” 혹은 “동적 그래프”라는 표현이 (Paszke 기타 2019년) 뜻하는 것이 바로 이것이며, 순전파 내부의 if 문이 예제마다 다른 서브 그래프로 라우팅되어도 아무 특별한 장치 없이 동작하는 이유이기도 합니다.

그림 2 은 학습된 함수와 원래 신호를 비교하고, 그림 3 는 손실 곡선을 로그 스케일로 보여 줍니다.

with torch.no_grad():
    y_fit = (torch.tanh(x_data @ W1 + b1) @ W2 + b2)

fig, ax = plt.subplots(1, 2, figsize=(10, 3.6))
ax[0].scatter(x_data.numpy(), y_data.numpy(), s=8, color="#888",
              label="noisy data")
ax[0].plot(x_data.numpy(), y_true.numpy(), color="#2c3e50",
           lw=1.4, label="clean signal")
ax[0].plot(x_data.numpy(), y_fit.numpy(),  color="#c0392b",
           lw=1.8, label="autograd fit")
ax[0].set_title("fit"); ax[0].set_xlabel("x"); ax[0].set_ylabel("y")
ax[0].legend(frameon=False)

resid = (y_fit - y_true).squeeze().numpy()
ax[1].hist(resid, bins=25, color="#7f8c8d", edgecolor="white")
ax[1].axvline(0, color="#c0392b", lw=1.2)
ax[1].set_title("residual (fit − clean signal)")
ax[1].set_xlabel("residual")
plt.tight_layout(); plt.show()
fig, ax = plt.subplots(figsize=(6.6, 3.4))
ax.plot(losses, color="#2c3e50", lw=1.4)
ax.set_yscale("log")
ax.set_xlabel("step"); ax.set_ylabel("MSE loss")
plt.tight_layout(); plt.show()

학습된 함수는 원 신호에 가깝고, 잔차는 대략 잡음의 표준편차 수준에서 0을 중심으로 분포합니다. 이 과제에서 어떤 모델이든 도달할 수 있는 한계선입니다. 로그 스케일의 손실 곡선은 두 단계 학습 양상을 드러냅니다. 빠른 대략 적합과 그 뒤의 느린 정밀 조정입니다. 두 그림 어느 쪽도 autograd 자체에 대한 새로운 정보를 주지는 않지만, 리프 텐서와 backward(), no_grad() 만으로 이루어진 손수 짠 루프가 표준 학습 API의 완전한 대체가 된다는 사실을 확인시켜 줍니다. 표준 API 위에서 비표준적인 무엇을 짜기 전에 확인해 두고 싶은 지점이 바로 이 부분입니다.

조용히 잘못된 그래디언트를 만드는 함정들

대부분의 autograd 버그는 예외를 던지지 않습니다. 대신, 그럴싸해 보이지만 우리가 최소화하고 싶은 손실의 그래디언트가 아닌 값을 만들어 냅니다. 실무에서 마주치는 사고 대부분이 다음 네 가지 상황에서 나옵니다.

역전파에 필요한 텐서에 대한 in-place 연산. x.add_, x.mul_, 슬라이스 대입 (x[0] = ...) 같은 연산자는 텐서의 저장 공간을 그 자리에서 수정합니다. 만약 그 텐서가 역방향 야코비언 계산에 필요한, 예컨대 도함수가 자기 값에 의존하는 연산의 사전활성화 입력이었다면, PyTorch가 런타임 오류를 던집니다. 오류가 나지 않는다면, 그 in-place 연산이 역전파에서 값이 필요 없는 텐서에 일어난 것이고 이때는 안전합니다. 안전한 기본값은 함수형 스타일로 쓰는 것입니다(x += 1 대신 y = x + 1). in-place 연산에 손을 대는 것은 프로파일링으로 메모리 압박이 확인된 다음이 낫습니다.

의도치 않은 그래프 분리. .detach(), .numpy(), .item(), 그리고 텐서를 파이썬 스칼라로 감싸는 어떤 방식이든 텐서를 그래프에서 떼어 냅니다. 평가 블록에서는 원하는 동작이지만, 손실 안에서는 재앙입니다. 흔한 실패 사례는 목표값을 .numpy() 로 뽑은 통계로 정규화한 뒤 손실 안에서 빼는 경우입니다. 목표에는 그래디언트 경로가 없어야 하지만(맞음), 같은 트릭을 예측값에도 적용해 버리면 손실에도 그래디언트 경로가 사라져 모델이 아예 학습하지 않습니다.

zero_grad 를 잊음. 위에서 본 대로 그래디언트는 누적됩니다. 이는 미니배치를 여러 번 쌓아 큰 배치를 흉내 내는 그래디언트 누적처럼, 원할 때만 선택해서 활용해야 합니다. p.grad = None 이나 zero_grad() 를 빠뜨리면 지금까지 루프가 계산한 모든 그래디언트의 합이 넘어와, 학습이 폭발하거나 무너지는 방식이 학습률 문제처럼 보이게 됩니다.

이중 역전파와 그래프 보존. 기본적으로 그래프는 backward() 후 해제됩니다. 그래디언트 페널티처럼 그래디언트에 대한 그래디언트가 다시 필요한 고차 도함수를 계산하려면 create_graph=True 를, 같은 그래프에서 backward() 를 두 번 부르려면 retain_graph=True 를 넘겨야 합니다. 둘 다 메모리를 소모합니다. 의도적으로 쓸 때는 옳은 도구이고, 오류 메시지를 없애려는 답답함에서 즉흥적으로 쓸 때는 설계 실수의 신호입니다.

autograd의 비용, 그리고 no_grad 가 중요한 순간

그래프를 만드는 일 자체가 공짜는 아닙니다. 연산 하나가 실행될 때마다 grad_fn이 기록되고, 역전파에 필요한 입력의 참조가 유지되며, 그 입력이 회수되지 못하도록 붙잡혀 있습니다. 큰 행렬곱의 연산량에 비하면 이 오버헤드는 작지만, 같은 순전파를 추론용으로 여러 번 반복하면 누적됩니다. 대략적인 비교입니다.

import time

torch.manual_seed(0)
Wa = torch.randn(2048, 2048, requires_grad=True)
Wb = torch.randn(2048, 2048, requires_grad=True)
x_big = torch.randn(2048, 2048)

# warmup
for _ in range(2):
    (x_big @ Wa @ Wb).sum().backward()
    Wa.grad = None; Wb.grad = None

t0 = time.perf_counter()
for _ in range(5):
    z = (x_big @ Wa) @ Wb
    s = z.sum(); s.backward()
    Wa.grad = None; Wb.grad = None
t_train = (time.perf_counter() - t0) / 5

t0 = time.perf_counter()
with torch.no_grad():
    for _ in range(5):
        z = (x_big @ Wa) @ Wb
        s = z.sum()
t_infer = (time.perf_counter() - t0) / 5

print(f"forward + backward:     {t_train*1000:.1f} ms")
print(f"forward under no_grad:  {t_infer*1000:.1f} ms")
print(f"training / inference:   {t_train/t_infer:.2f}x")
forward + backward:     82.8 ms
forward under no_grad:  32.3 ms
training / inference:   2.56x

이 비율의 절댓값 자체는 하드웨어·자료형·연산 조합에 따라 달라지므로 그 자체가 결론은 아닙니다. 그러나 답의 모양은 분명해집니다. 학습은 역전파의 비용과 중간 활성화를 살려 두는 비용을 함께 지불하고, no_grad 아래의 추론은 그 두 비용을 모두 건너뜁니다. 위 크기의 CPU 실행에서는 몇 배 수준이지만, GPU에서 어텐션이 섞이면 한 자릿수 배로 커질 수 있으며, 원인 모를 추론 서버 지연을 마주할 때 가장 먼저 의심해야 할 지점입니다.

autograd가 답이 아닌 상황

미분 불가능한 연산에는 그래디언트가 흐르지 않습니다. torch.argmax, 정수 형변환, 임계값 마스크, 제어 흐름 기반 이산 결정은 모두 그래프를 끊습니다. 이산 선택에는 Gumbel-Softmax, 양자화에는 straight-through estimator 같은 미분 가능한 대체물을 끼워 넣어 우회하는 방법이 있지만, 우회를 하고 있다는 사실을 알고 해야 하지, 그래디언트가 조용히 0이 되어서야 발견할 일은 아닙니다. 둘째, 메모리는 유한합니다. 아주 깊은 그래프 (길게 펼친 RNN, 여러 스텝을 적분하는 확산 과정)는 저장된 활성화까지 함께 담기지 않을 수 있고, torch.utils.checkpoint 의 그래디언트 체크포인팅은 이 지점에서 계산과 메모리를 맞바꿉니다. 셋째, 이차 정보의 일부는 여전히 비쌉니다. Hessian-vector 곱은 역전파를 다시 한 번 역전파해 효율적으로 얻을 수 있지만, 백만 파라미터짜리 모델의 완전 Hessian은 계획이 아니라 소망입니다. 마지막으로, autograd는 우리가 실제로 쓴 코드의 도함수를 계산합니다. 우리가 쓴 손실이 우리가 신경 쓰는 것을 재지 못한다면, autograd는 잘못된 목표를 충실히 최소화할 뿐입니다.

결론

autograd는 마법이 아닙니다. 텐서 연산이 조립한 동적 DAG이고, 각 노드가 스칼라 출력에 대해 연쇄 법칙을 역방향으로 실행하기에 충분한 정보를 담고 있는 구조체입니다. 이 관점을 가져야 얻을 수 있는 실무적 이득이 셋 있습니다. 첫째, gradcheck 로 검증할 수 있으므로 커스텀 손실과 층을 두려움 없이 쓸 수 있습니다. 둘째, 역전파 비용과 활성화 보존 비용의 정체를 알게 되므로 학습 루프의 성능 프로파일을 읽을 수 있습니다. 셋째, 조용한 0, 폭발하는 그래디언트, 끝없이 자라는 메모리처럼 원인 모를 학습 실패가 결국 위의 함정 중 하나로 귀착된다는 사실을 알므로, 이런 실패를 진단할 수 있습니다. 다음 편은 이 토대 위에서 데이터 파이프라인으로 넘어갑니다. 시계열용 슬라이딩 윈도우 DataLoader 를 손으로 구현하는데, 그래프 구성에 대한 같은 걱정이 이번에는 배치와 셔플과 얽혀 다시 등장합니다.
