모두가 건너뛰는 층

모든 LLM 벤치마크, 프롬프트 엔지니어링 요령, 스케일링 법칙 그래프는 거의 언급되지 않는 전처리 단계 위에 서 있습니다. 토크나이저입니다. 모델이 임베딩 하나를 만들기도 전에 고정된 알고리즘이 이미 원문을 고정 어휘의 정수 시퀀스로 잘라 놓았고, 그 자르는 방식이 모델이 볼 수 있는 것을 결정합니다. 같은 뜻이지만 띄어쓰기나 구두점이 다른 두 문장은 서로 다른 토큰 시퀀스가 되어, 어텐션 패턴도 퍼플렉시티도 학습 신호도 달라집니다. 2026-07-15 같은 숫자는 모델에게 하나의 개체가 아닙니다. 통계량이 토크나이저 학습 코퍼스에 우연히 등장한 텍스트에 의해 결정된 서브워드 조각들의 시퀀스입니다.

수치·과학 데이터를 LLM에 태우는 사람에게 — 예측, 코드, 표, 금융 시계열 — 토크나이저는 배경 세부사항이 아닙니다. 시스템적 오차가 처음 주입되는 지점이고, 모델 층에서 되돌리려 하면 비싸집니다. 이 글은 그 토크나이저를 엽니다. Byte Pair Encoding이 실제로 무엇을 하는지 작은 코퍼스로 처음부터 구현해 병합 규칙을 눈에 보이게 만들고, WordPiece와의 차이를 짚고, 숫자 문자열이 하위 작업 성능을 해치는 방식으로 쪼개지는 이유를 살펴봅니다. NLP에서 쓰이는 BPE 알고리즘은 Sennrich 기타 (2016년) 이 제안했고, WordPiece는 Schuster 와/과 Nakajima (2012년) 로 거슬러 올라가며 BERT의 채택은 Devlin 기타 (2019년) 입니다. 숫자 토큰화의 병리는 Thawani 기타 (2021년) 과 Nogueira 기타 (2021년) 이 실증했습니다.

토크나이저가 결정하는 것

문자열이 주어지면 토크나이저는 정수 시퀀스 \(x_1, \dots, x_n\) 를 내놓습니다. 각 \(x_i \in \{1, \dots, V\}\) 는 크기 \(V\) 의 고정 어휘에 대한 색인입니다. 이 매핑의 세 가지 속성이 하위 동작에 영향을 줍니다.

\[ \text{coverage:}\ \Pr[\text{unk}] = 0, \qquad \text{fertility:}\ f(s) = \frac{|\text{tokenize}(s)|}{|s_{\text{words}}|}, \qquad \text{alignment:}\ \text{tokenize}(s_1) \cap \text{tokenize}(s_2)\ \text{when}\ s_1 \simeq s_2. \tag{1}\]

커버리지는 자연 입력에 대해 알 수 없는 토큰(unk)을 낼 확률입니다. 현대 토크나이저가 쓸모 있으려면 이 값이 0이어야 하고, 바이트 수준 방식은 이를 구성적으로 보장합니다. 퍼틸리티는 입력 단어당 평균 토큰 수로, 실효 컨텍스트 길이와 문서당 연산 비용을 동시에 결정합니다. 퍼틸리티 1.4인 토크나이저는 1.0인 토크나이저보다 문서당 40% 더 비쌉니다. 얼라인먼트는 의미가 같은 입력이 겹치는 서브시퀀스로 토큰화되어 모델이 학습된 표현을 재사용할 수 있어야 한다는 성질이고, 토크나이저의 공백·구두점 정책이 이를 좌우합니다.

단어 수준 토큰화는 셋 다 실패합니다. 어휘가 폭발하거나 <unk> 를 많이 냅니다. 문자 수준 토큰화는 커버리지 1과 작은 \(V\) 를 얻는 대신 퍼틸리티가 폭발해, 모델이 처음부터 다시 학습해야 하는 형태학적 규칙성에 어텐션 예산을 낭비합니다. 서브워드 토큰화(BPE, WordPiece, Unigram)는 이 절충으로 자리 잡은 기본값입니다 (Bostrom 와/과 Durrett 2020년).

머릿속에 들어오는 코퍼스로 BPE를 처음부터

BPE 알고리즘은 몇십 줄로 구현할 만큼 단순합니다. 코퍼스에서 단어 빈도를 세고, 각 단어를 문자 시퀀스(끝에 단어 종료 표시)로 쪼갠 다음, 코퍼스 전체에서 가장 빈번한 인접 쌍을 찾아 하나의 토큰으로 병합하는 일을 반복합니다. 각 병합은 기록되며, 추론 시 같은 병합이 같은 순서로 어떤 입력에도 적용됩니다. 병합의 목록이 곧 토크나이저입니다.

from collections import Counter, defaultdict

corpus = [
    "the model predicts sales",
    "the model predicts demand",
    "the model predicts prices",
    "modeling sales and demand",
    "modeling prices and volumes",
    "predicting sales volumes",
    "predicting demand shocks",
]

def word_freqs(corpus):
    c = Counter()
    for line in corpus:
        for w in line.lower().split():
            c[w] += 1
    return c

def make_splits(freqs):
    return {w: list(w) + ["</w>"] for w in freqs}

def get_pair_stats(splits, freqs):
    stats = defaultdict(int)
    for w, split in splits.items():
        for i in range(len(split) - 1):
            stats[(split[i], split[i + 1])] += freqs[w]
    return stats

def merge_pair(pair, splits):
    a, b = pair
    new = {}
    for w, split in splits.items():
        out, i = [], 0
        while i < len(split):
            if i < len(split)-1 and split[i]==a and split[i+1]==b:
                out.append(a+b); i += 2
            else:
                out.append(split[i]); i += 1
        new[w] = out
    return new

freqs  = word_freqs(corpus)
splits = make_splits(freqs)
vocab  = set(ch for w in splits.values() for ch in w)
merges, sizes = [], [len(vocab)]

for _ in range(12):
    stats = get_pair_stats(splits, freqs)
    if not stats: break
    best = max(stats, key=stats.get)
    splits = merge_pair(best, splits)
    merges.append(best)
    vocab.add(best[0]+best[1])
    sizes.append(len(vocab))

print("first 8 merges:")
for i, m in enumerate(merges[:8], 1):
    print(f"  {i:2d}. {m[0]!r} + {m[1]!r} -> {(m[0]+m[1])!r}")

print("\nfinal splits (first 5 words):")
for w in list(splits.keys())[:5]:
    print(f"  {w:12s} -> {splits[w]}")
first 8 merges:
   1. 's' + '</w>' -> 's</w>'
   2. 'd' + 'e' -> 'de'
   3. 'p' + 'r' -> 'pr'
   4. 'i' + 'c' -> 'ic'
   5. 'e' + 's</w>' -> 'es</w>'
   6. 'm' + 'o' -> 'mo'
   7. 'mo' + 'de' -> 'mode'
   8. 'mode' + 'l' -> 'model'

final splits (first 5 words):
  the          -> ['t', 'h', 'e', '</w>']
  model        -> ['model', '</w>']
  predicts     -> ['predict', 's</w>']
  sales        -> ['s', 'a', 'l', 'es</w>']
  demand       -> ['de', 'm', 'a', 'n', 'd', '</w>']

병합 목록이 읽힙니다. 첫 두 병합 — 's' + '</w>' 와 'd' + 'e' — 은 이 코퍼스에서 가장 흔한 바이그램을 포착합니다. 복수형 접미사와, 실제 코퍼스에서 demand, deliver, decrease 등 여러 어휘가 시작되는 “de” 입니다. 여덟 번째 병합에 이르면 'model' 이 하나의 토큰이 됩니다. 자주 등장하고, 그 서브워드 조각들('mo', 'de', 'l')이 차례로 병합되었기 때문입니다. 이 상향식 구성은 Sennrich 기타 (2016년) 이 주장한 성질입니다. 빈번한 문자열은 원자가 되고, 드문 문자열은 가장 재사용 가능한 조각들로 분해되며, 둘 사이의 경계는 임의의 규칙이 아니라 데이터가 결정합니다.

그림 1 은 병합이 누적됨에 따라 어휘 크기가 한 번에 하나씩 늘어나는 모습을 보여 줍니다. 이 선형 성장은 겉보기와 달리 단순한 이야기가 아닙니다. 각 병합은 토큰을 삭제하지 않고(병합되지 않는 단어들의 원자는 여전히 남습니다) 새 토큰을 하나 추가합니다. 실제 학습에서는 목표 어휘 크기에서 멈추고, 마지막 병합들이 가장 정보성이 높은 경향이 있습니다. 드물지만 반복되는 형태학 패턴(-tion</w>, -ing</w> 같은)을 포착하며, 미지 텍스트에서의 퍼틸리티에 가장 큰 영향을 주는 병합들입니다.

WordPiece: 같은 아이디어, 다른 목적함수

BPE는 공기 빈도 를 최대화하는 병합을 고릅니다. BERT (Devlin 기타 2019년) 가 채택했고 원래는 일본어·한국어 음성 인식용으로 제안된 (Schuster 와/과 Nakajima 2012년) WordPiece는 유니그램 언어모델 아래에서 학습 코퍼스의 우도 를 최대화하는 병합을 고릅니다. 토큰 \(a, b\) 의 빈도가 각각 \(c(a), c(b)\) 이고 쌍 \(ab\) 의 빈도가 \(c(ab)\) 일 때, WordPiece가 고르는 쌍은

\[ \frac{c(ab)}{c(a)\, c(b)} \tag{2}\]

를 최대화하는 쌍입니다. 각 조각이 원래 얼마나 흔한지로 원생 쌍 빈도를 정규화합니다. 실질적 효과는, 조각이 흔하다는 이유로 흔한 쌍(t + h 같은)을 공격적으로 병합하지 않고, 공기가 특별히 높은 쌍을 병합하는 데 더 적극적이 되는 것입니다. WordPiece는 또한 비-어두 서브워드를 접두 표시(##ing, ##tion)로 인코딩해 토큰화가 자명하게 단어로 가역 복원되도록 합니다. 같은 코퍼스로 학습하면 두 방식은 구조적으로 비슷하지만 동일하지 않은 분할을 만듭니다.

from tokenizers import Tokenizer
from tokenizers.models import BPE, WordPiece
from tokenizers.pre_tokenizers import ByteLevel, Whitespace
from tokenizers.trainers import BpeTrainer, WordPieceTrainer

sentences = [
    "the model predicts sales", "the model predicts demand",
    "the model predicts prices", "modeling sales and demand",
    "modeling prices and volumes", "predicting sales volumes",
    "predicting demand shocks",
    "the price rose from 100 to 105 dollars",
    "sales grew from 12345 to 12987 units",
    "the model predicts 3.14 percent growth",
    "temperature dropped from 27.5 to 22.3 celsius",
    "demand shifted by 0.05 standard deviations",
    "the forecast for tomorrow is warm and dry",
    "the model uses historical data and current features",
    "predicting the future requires assumptions about the past",
]
train_text = sentences * 300

bpe = Tokenizer(BPE())
bpe.pre_tokenizer = ByteLevel(add_prefix_space=True)
# 학습 코퍼스에 등장하지 않은 바이트(예: 이 예제에 없는 자릿수 '6')도 어딘가로 매핑되도록
# 256개 바이트 전체를 초기 알파벳에 강제로 포함시킵니다.
bpe.train_from_iterator(train_text, BpeTrainer(
    vocab_size=1500, special_tokens=["<pad>"],
    initial_alphabet=ByteLevel.alphabet()))

wp = Tokenizer(WordPiece(unk_token="[UNK]"))
wp.pre_tokenizer = Whitespace()
wp.train_from_iterator(train_text, WordPieceTrainer(
    vocab_size=1500, special_tokens=["[PAD]", "[UNK]"],
    continuing_subword_prefix="##"))

def show(ts): return [t.replace("\u0120", "_") for t in ts]

for ex in ["predicts", "predicting", "modeling", "12345", "unseen"]:
    print(f"{ex!r:14s}  BPE: {show(bpe.encode(ex).tokens)}"
          f"    WordPiece: {wp.encode(ex).tokens}")






'predicts'      BPE: ['_predicts']    WordPiece: ['predicts']
'predicting'    BPE: ['_predicting']    WordPiece: ['predicting']
'modeling'      BPE: ['_modeling']    WordPiece: ['modeling']
'12345'         BPE: ['_12345']    WordPiece: ['12345']
'unseen'        BPE: ['_', 'u', 'ns', 'e', 'e', 'n']    WordPiece: ['un', '##se', '##e', '##n']

두 토크나이저 모두 학습에서 흔했던 단어는 통째로 유지합니다. 흥미로운 행은 unseen 입니다. 학습에 없었으므로 두 알고리즘 모두 분해하되, 원자가 다릅니다. WordPiece는 ## 로 연속을 표시해 시퀀스가 명확하게 재조합되고, 바이트 수준 BPE는 어두 위치를 표시합니다(위 출력의 _ 가 공백 표시입니다). 재조합의 방향이 반대이지만 둘 다 명확 합니다. 이 세부가 중요한 이유는, 하위 태스크가 구간에 민감할 때(질의응답, 개체명 인식) 같은 문자 범위에 대해 토크나이저가 다르면 다른 구간 경계를 주기 때문입니다.

숫자 문제

예측에 가장 심하게 부딪히는 병리는 토크나이저가 숫자에 하는 일입니다. 자연어 코퍼스는 길고 고유한 숫자 문자열을 상대적으로 적게 포함하기 때문에, 대부분의 BPE·WordPiece 토크나이저는 아주 흔한 것(반올림한 백 단위, 연도) 정도를 제외하면 다자릿수 숫자 서브워드를 학습하지 못합니다. 그 결과 1234.56 같은 숫자는 하나의 항목으로 표현되지 않고, 통계량이 사실상 개별 자릿수의 그것에 가까운 자릿수 조각 토큰들의 시퀀스가 됩니다 (Thawani 기타 2021년).

examples = ["price 100", "price 1234.56", "17.85",
            "2026-07-15", "$3.14"]
for ex in examples:
    t = bpe.encode(ex).tokens
    print(f"{ex!r:16s} -> {show(t)}   ({len(t)} tokens)")
'price 100'      -> ['_price', '_100']   (2 tokens)
'price 1234.56'  -> ['_price', '_1234', '.', '5', '6']   (5 tokens)
'17.85'          -> ['_1', '7', '.', '8', '5']   (5 tokens)
'2026-07-15'     -> ['_2', '0', '2', '6', '-', '0', '7', '-', '1', '5']   (10 tokens)
'$3.14'          -> ['_', '$', '3', '.', '14']   (5 tokens)

출력이 전체 이야기를 말해 줍니다. price 100 은 학습 코퍼스에 “100” 이 자주 나와 두 토큰입니다. price 1234.56 은 다섯 토큰으로 쪼개집니다. “1234” 는 서브워드로 봤지만 소수부 .56 은 본 적이 없고, 그 개별 자릿수 각각이 별도의 토큰이 됩니다. 2026-07-15 같은 날짜는 전체 문자열이 열 자에 불과한데도 열 개의 토큰으로 쪼개집니다. 하이픈 하나 하나와 연도의 개별 자릿수가 각각 토큰 하나씩을 소비합니다. 하위 모델이 실제로 겪는 결과는 시퀀스가 길어지는 것에 그치지 않습니다. 정체성이 기저의 수 구조를 거의 담지 못하는 기호에 대해 모델이 산술을 학습해야 한다는 것입니다. 숫자는 크기이지만, 토큰은 기호입니다. 토크나이저는 그 구분을 뭉갰고, 모델이 다시 세워야 합니다.

그림 2 는 이 성장을 명시적으로 보입니다. 자릿수가 늘수록 토큰 수는 대략 선형으로 늘어납니다. 정확한 기울기는 어떤 짧은 자릿수 문자열이 우연히 어휘에 들어갔는지에 따라 달라집니다.

digits = list(range(1, 13))
counts = [len(bpe.encode("".join(str(i % 10) for i in range(d))).tokens)
          for d in digits]
print("digits -> token count:")
for d, c in zip(digits, counts):
    print(f"  {d:2d} digits: {c} tokens")

fig, ax = plt.subplots(figsize=(6.4, 3.4))
ax.plot(digits, counts, marker="o", color="#c0392b", lw=1.6)
ax.plot(digits, digits, color="#7f8c8d", lw=1.0, linestyle="--",
        label="1 token per digit (upper bound)")
ax.set_xlabel("digits in input"); ax.set_ylabel("tokens produced")
ax.legend(frameon=False); ax.grid(alpha=0.3)
plt.tight_layout(); plt.show()

예측 관점의 파장은 구체적입니다. Nogueira 기타 (2021년) 은 GPT-3의 다자릿수 덧셈 정확도가 피연산자가 어떻게 토큰화되는지에 크게 의존한다는 것을 보였습니다. 자릿수 단위 토큰화가 기본 BPE보다 실질적으로 더 나은 체계적 일반화를 얻었습니다. 이후 연구는 두 가지 대응을 탐색했습니다. 토크나이저를 항상 자릿수당 한 토큰을 내도록 고치는 방식(예: LLaMA의 수치 토큰화 정책)과, 수치와 텍스트 스트림을 완전히 분리해 공유 토큰 임베딩이 아니라 전용 사영을 통해 숫자를 임베딩하는 방식입니다. 두 접근 모두, 산문을 위한 통계 기계로 숫자를 같게 다루는 것은 스케일만으로 풀 문제가 아니라 설계상의 오류라는 인식을 공유합니다.

실무에서 드러나는 결과

위의 기제에서 세 가지 실무적 문제가 뒤따라 나옵니다.

비용과 컨텍스트 길이. 퍼틸리티가 1.0 대비 1.3인 토크나이저는 같은 모델 품질에서 쓸모 있는 컨텍스트가 30% 줄고 문서당 추론 비용은 30% 늘어납니다. 긴 컨텍스트 응용 (법률 문서, 코드베이스, 논문)에서는 이 오버헤드가 태스크가 컨텍스트 창에 들어가느냐를 결정합니다. 다국어 토크나이저는 이를 더 악화시킵니다. 영어에 최적화된 BPE는 학습에 과소 대표된 문자(태국어, 벵골어, 조지아어)에서 대개 3–5배 높은 퍼틸리티를 보이며, 그 언어들에서 LLM 품질이 가파르게 떨어지는 이유의 일부입니다 (Petrov 기타 2023년).

프롬프트 민감성. 사람에게는 동일해 보이는 두 프롬프트가 — "1.5%" 대 "1.5 %" 같은 — 완전히 다른 시퀀스로 토큰화될 수 있습니다. 공백, 대소문자, 구두점의 작은 편집이 모델을 서로 다른 어텐션 패턴과 다른 상위-\(k\) 예측을 가진 임베딩 공간 영역으로 이동시킵니다. “프롬프트 엔지니어링은 마법 같다” 는 불평의 상당수는 모델을 지나 토크나이 저까지 거슬러 올라갑니다.

구조화된 입력과의 얼라인먼트. LLM에 구조화된 데이터(JSON, CSV, 코드, 화학 구조식) 를 넣을 때, 산문용 토크나이저는 그 구조를 존중하지 않는 경계를 도입합니다. JSON 키가 토큰 경계에서 쪼개지고, 부동소수가 소수점에서 쪼개지고, 함수명이 밑줄에서 쪼개집니다. 코드 특화 토크나이저(StarCoder, Code Llama)는 다른 프리토큰화 규칙으로 학습된 코드 코퍼스를 사용해 이를 완화합니다. 과학·수치 표 데이터에는 선택지가 훨씬 적습니다.

토크나이저 층에서 개입할 때

모든 프로젝트가 커스텀 토크나이저를 만들어야 하는 것은 아닙니다. 토크나이저를 다시 학습한다는 것은 그것을 읽는 모델을 다시 학습한다는 뜻이며, 어떤 규모에서든 그 비용은 현실적으로 큽니다. 그러나 세 상황은 물려받은 토크나이저가 무엇인지 최소한 검토할 이유가 됩니다.

첫째, 수치 정확도가 일차 요구인 경우 — 산술, 원생 수치 문자열로부터의 예측, 숫자를 다루는 코드 — 자릿수 단위 토큰화를 갖는 모델을 쓰거나, 모델이 보기 전 일관된 형태로 변환되도록 수치 입력을 감싸는 것이 엔지니어링 시간을 들일 값어치가 있습니다. 둘째, 대상 도메인이 저자원 언어이거나 특수 문자 체계(화학, 수학, 법률 표기)라면, 기본 토크나이저의 그 도메인에 대한 퍼틸리티가 사후 조치로 올릴 수 없는 비용-효율의 상한을 정합니다. 셋째, 하위 태스크가 문자 단위 통제를 요구할 때 — 철자 교정, 토큰화-불변 검색, 구간 단위 주석 — 서브워드 토크나이저는 예측 불가능하게 표면화되는 버그의 원천이 되고, 바이트 수준 운영이나 명시적 문자 수준 경로가 더 안전한 선택입니다.

결론

토큰화는 모델러가 남의 문제로 두어도 되는 전처리가 아닙니다. 모델이 조작할 수 있는 어휘를 고정하고, 컨텍스트와 비용을 결정하는 퍼틸리티를 세팅하며, 예측에 가장 결정적 이게도 숫자에 대해 그 구조를 버리는 기호 표현을 강제합니다. BPE와 WordPiece는 산문에 대해서는 원칙적인 선택이지만, 숫자·날짜·구조화된 입력에서의 행동은 스케일만으로는 아직 고치지 못한 버그 유발 코너입니다. 프로젝트가 관심 있는 입력 분포에 대해 토크나이저 출력을 감사하는 일은, 놀랄 만큼 많은 놀람을 잡아내는 값싼 점검입니다. 다음 LLM 편은 이 이야기를 구체적 아키텍처 문제로 이어 갑니다. 위치 인코딩 체계 — sinusoidal, RoPE, ALiBi — 가 토크나이저가 방금 결정한 시퀀스 길이와 어떻게 상호작용하는지 다룹니다.
