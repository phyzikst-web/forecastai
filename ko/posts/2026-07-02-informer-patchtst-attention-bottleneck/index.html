셀프 어텐션은 시퀀스 길이에 대해 O(L^2) 비용이 들며, 이는 긴 시계열에 치명적입니다. 이 글은 그 비용을 벗어나는 매우 다른 두 경로 — Informer의 ProbSparse 어텐션과 PatchTST의 패칭 — 을, 절감이 실제로 어디서 나오는지 측정하는 실행 가능한 코드와 함께 살펴봅니다.

1. 길이가 곧 적이다

트랜스포머(Transformer)는 모든 위치가 다른 모든 위치를 참조하게 함으로써 정확도를 얻습니다. 길이 \(L\), 모델 차원 \(d\)인 시퀀스에서 이는 \(L \times L\) 점수 행렬을 만든다는 뜻이고, 그 연산량과 메모리는 모두 \(O(L^2 d)\)로 늘어납니다. 언어에서는 문장이 짧아 견딜 만합니다. 그러나 장기 시계열 예측(long-sequence time series forecasting, LSTF)에서는 치명적입니다. 수백~수천 스텝의 look-back 윈도우를 가진 전력·교통 시계열에서는 어텐션 행렬이 지배적인 비용이 되고, 메모리 사용량만으로도 GPU가 담을 수 있는 한계를 넘어설 수 있습니다 (Zhou 기타 2021년).

영향력 있는 두 모델이 이 문제를 정반대 방향에서 공략했습니다. Informer는 트랜스포머의 토큰 구조를 유지하되 중요한 쿼리에 대해서만 어텐션을 계산하여 비용을 \(O(L \log L)\)로 줄입니다 (Zhou 기타 2021년). PatchTST는 완전한 어텐션을 유지하되 먼저 시퀀스를 짧게 만듭니다. 원시 시간 스텝들을 패치(patch)로 묶어 참조해야 할 토큰 수 자체를 크게 줄입니다 (Nie 기타 2023년). 앞은 \(L \times L\) 행렬을, 뒤는 \(L\)을 공격합니다. 각 절감이 정확히 어디서 나오는지는 코드로 볼 때 가장 분명하므로, 이 글은 설명만 늘어놓기보다 작은 실행 실험에 기댑니다.

먼저 병목을 구체적으로 만들어 봅니다.

import torch
import torch.nn as nn
import math, time
torch.manual_seed(0)

def dense_attention(x):
    """교과서적인 스케일드 닷-프로덕트 셀프 어텐션.
    x: (B, L, d) -> (B, L, d)와 L x L 점수 행렬을 반환한다."""
    B, L, d = x.shape
    scores = (x @ x.transpose(1, 2)) / math.sqrt(d)   # (B, L, L)  <-- O(L^2) 대상
    attn = torch.softmax(scores, dim=-1)
    out = attn @ x                                    # (B, L, d)
    return out, scores

for L in [96, 336, 720]:
    x = torch.randn(1, L, 64)
    _, scores = dense_attention(x)
    n_entries = scores.numel()
    mem_mb = scores.element_size() * n_entries / 1e6
    print(f"L={L:4d} -> score matrix {tuple(scores.shape)}, "
          f"{n_entries:,} entries, ~{mem_mb:.2f} MB (fp32) for ONE head")
L=  96 -> score matrix (1, 96, 96), 9,216 entries, ~0.04 MB (fp32) for ONE head
L= 336 -> score matrix (1, 336, 336), 112,896 entries, ~0.45 MB (fp32) for ONE head
L= 720 -> score matrix (1, 720, 720), 518,400 entries, ~2.07 MB (fp32) for ONE head

점수 행렬은 \(L\)의 제곱으로 늘어납니다. 96 스텝 윈도우에서 720 스텝 윈도우로 가면 크기가 56배 넘게 커집니다. 여기에 헤드 수와 레이어 수를 다시 곱하면, 긴 윈도우가 왜 오랫동안 다루기 어려웠는지 분명해집니다.

2. Informer: 실제로 중요한 쿼리는 몇 개뿐이다

Informer의 출발점은 경험적 관찰입니다. 학습된 트랜스포머의 어텐션 분포를 보면, 대부분의 쿼리 행은 거의 균일합니다. 모든 키에 가중치를 고루 흩뿌려 출력에 별로 기여하지 않습니다. 반면 소수의 쿼리 행만 날카롭게 뾰족하여 출력을 지배합니다 (Zhou 기타 2021년). 따라서 전체 \(L \times L\) 행렬을 계산하는 것은, 결과를 거의 바꾸지 않는 행들에 대부분의 노력을 낭비하는 셈입니다.

Informer는 이를 희소성 측도로 정밀화합니다. 쿼리 \(i\)의 어텐션 “뾰족함”은 다음과 같습니다.

\[ M(q_i, K) = \max_j \frac{q_i k_j^\top}{\sqrt{d}} - \frac{1}{L}\sum_{j=1}^{L}\frac{q_i k_j^\top}{\sqrt{d}}, \tag{1}\]

쿼리의 최대 점수와 평균 점수의 차이입니다. \(M\)이 크면 그 쿼리는 소수의 키에 집중하므로 정확히 계산할 가치가 있고, \(M\)이 작으면 거의 균일하므로 값싼 평균으로 대체할 수 있습니다. 여기서 \(q_i \in \mathbb{R}^d\)는 하나의 쿼리, \(K \in \mathbb{R}^{L \times d}\)는 키 행렬, \(k_j\)는 그 \(j\)번째 행입니다. 모든 쿼리에 대해 방정식 1 를 계산하면 이미 \(O(L^2)\)이 들므로, Informer는 무작위로 뽑은 \(U = c \ln L\)개의 키만으로 이를 추정한 뒤 상위 \(u = c \ln L\)개의 쿼리만 남깁니다. 나머지는 모두 평균으로 대체됩니다. 그 결과가 시간·메모리 \(O(L \log L)\)의 ProbSparse 어텐션입니다 (Zhou 기타 2021년).

이 메커니즘은 직접 구현할 만큼 짧습니다.

def probsparse_selection(Q, K, c=5):
    """eq-sparsity의 희소성 측도로 상위 u개 '활성' 쿼리를 선택한다.
    키의 부분집합을 샘플링해 측도를 추정한다.
    Q, K: (L, d). 활성 쿼리 인덱스와 밀집 계산 비율을 반환한다."""
    L, d = Q.shape
    scale = 1.0 / math.sqrt(d)
    U = min(L, int(c * math.log(L)))       # 샘플링하는 키 개수
    u = min(L, int(c * math.log(L)))       # 남기는 활성 쿼리 개수
    idx = torch.randperm(L)[:U]
    S_sample = (Q @ K[idx].T) * scale      # (L, U) 샘플 점수
    M = S_sample.max(dim=1).values - S_sample.mean(dim=1)   # eq-sparsity 근사
    active = torch.topk(M, u).indices
    return active, U, u

torch.manual_seed(0)
for L in [96, 336, 720]:
    Q = torch.randn(L, 32); K = torch.randn(L, 32)
    active, U, u = probsparse_selection(Q, K)
    dense = L * L
    sparse = u * L
    print(f"L={L:4d}: sampled keys U={U:2d}, active queries u={u:2d} "
          f"({u/L:5.1%} of queries) -> attention entries {dense:,} -> {sparse:,} "
          f"({dense/sparse:.1f}x fewer)")
L=  96: sampled keys U=22, active queries u=22 (22.9% of queries) -> attention entries 9,216 -> 2,112 (4.4x fewer)
L= 336: sampled keys U=29, active queries u=29 ( 8.6% of queries) -> attention entries 112,896 -> 9,744 (11.6x fewer)
L= 720: sampled keys U=32, active queries u=32 ( 4.4% of queries) -> attention entries 518,400 -> 23,040 (22.5x fewer)

출력은 이득과 함정을 함께 보여 줍니다. \(L=720\)에서는 쿼리의 몇 퍼센트만 전체 키에 대해 계산되어 어텐션 항목 수가 한 자릿수 배수로 줄어듭니다. 그런데 활성 쿼리 개수는 \(L\)이 아니라 \(\ln L\)처럼 늘어나며, 이것이 바로 점근 비용이 \(O(L \log L)\)로 떨어지는 이유입니다. 함정은 이 선택이 확률적이라는 점입니다. 무작위 키 샘플링에 의존하므로 근사 품질이 상수 \(c\)와, 어텐션이 실제로 롱테일인지 여부에 달려 있습니다. 어텐션이 희소하지 않은 시계열에서는 같은 기법이 정보를 버립니다.

Informer는 어텐션 행렬 자체와는 무관하지만 실무에서 중요한 두 가지를 더합니다. 인코더 레이어 사이에서 시퀀스 길이를 절반으로 줄여 깊은 레이어일수록 점점 짧은 입력을 보게 하는 셀프 어텐션 디스틸링(self-attention distilling), 그리고 예측을 스텝별로 내지 않고 한 번의 순전파로 전체를 내보내 자기회귀 디코딩의 오차 누적을 피하는 생성형 디코더(generative decoder)입니다 (Zhou 기타 2021년).

3. PatchTST: 어텐션 전에 시퀀스를 짧게 만든다

PatchTST는 다른 데 승부를 겁니다. 어텐션을 근사하는 대신 토큰의 개수를 줄입니다. 원시 시계열은 순진한 트랜스포머에서 한 스텝당 한 토큰으로 다뤄지는데, 한 시점은 그 자체로 담는 의미가 적습니다. 단어 하나보다 훨씬 적습니다. PatchTST는 연속된 스텝을 패치로 묶고, 각 패치를 벡터로 임베딩하여 하나의 토큰으로 넣습니다 (Nie 기타 2023년). 패치 길이 \(P\), 스트라이드 \(S\)일 때, 길이 \(L\)의 시계열은 \(L\)개가 아니라 대략 \(L/S\)개의 토큰을 만듭니다.

어텐션은 토큰 수에 대해 이차이므로, 토큰을 \(S\)로 나누면 어텐션 비용은 약 \(S^2\)배 줄어듭니다. 패칭 자체는 단순한 슬라이딩 윈도우입니다.

def patchify(x, P, S):
    """단변량 시계열을 겹치는 패치들로 나눈다.
    x: (B, L) -> (B, n_patches, P), n_patches ~ (L - P)/S + 1."""
    patches = x.unfold(dimension=1, size=P, step=S)   # (B, n_patches, P)
    return patches

B, L = 4, 336
x = torch.randn(B, L)
for (P, S) in [(16, 8), (16, 16), (24, 24)]:
    p = patchify(x, P, S)
    n = p.shape[1]
    print(f"L={L}, patch P={P}, stride S={S} -> {n} tokens "
          f"(vs {L} raw steps, {L/n:.1f}x fewer tokens)")
L=336, patch P=16, stride S=8 -> 41 tokens (vs 336 raw steps, 8.2x fewer tokens)
L=336, patch P=16, stride S=16 -> 21 tokens (vs 336 raw steps, 16.0x fewer tokens)
L=336, patch P=24, stride S=24 -> 14 tokens (vs 336 raw steps, 24.0x fewer tokens)

토큰이 적다는 것은 패치가 낱개 스텝보다 나은 의미 단위일 때에만 유용하며, 이것이 PatchTST의 두 번째 주장입니다. 패치는 낱점이 담지 못하는 국소 의미 정보 — 짧은 형태, 작은 추세 — 를 보존하므로, 모델은 더 적은 토큰에 어텐션하면서 동시에 더 의미 있는 토큰에 어텐션합니다 (Nie 기타 2023년). 논문은 패칭을 채널 독립성(channel independence) 과 결합합니다. 다변량 시계열의 각 변수를 초기에 섞지 않고 동일한 공유 단변량 모델로 처리하며, 이것이 채널을 섞는 설계보다 더 강건하다고 보고합니다 (Nie 기타 2023년).

이제 핵심 비교입니다. 시퀀스를 짧게 만드는 것이 실제로 무엇을 사 주는가? 이론적 어텐션 비용과 실측 시간을 모두 측정합니다.

def attn_cost(n_tokens, d):
    """지배적 어텐션 FLOPs: QK^T 다음 (softmax)V, 각각 ~ n^2 * d."""
    return 2 * n_tokens * n_tokens * d

d, P, S = 64, 16, 16
print("theoretical attention cost, full (L tokens) vs patched (L/S tokens)")
print(f"{'L':>5} {'full':>14} {'patched':>12} {'ratio':>8}")
for L in [96, 336, 720, 1440]:
    n_patch = (L - P) // S + 1
    full = attn_cost(L, d)
    patched = attn_cost(n_patch, d)
    print(f"{L:5d} {full:14,} {patched:12,} {full/patched:7.1f}x")
theoretical attention cost, full (L tokens) vs patched (L/S tokens)
    L           full      patched    ratio
   96      1,179,648        4,608   256.0x
  336     14,450,688       56,448   256.0x
  720     66,355,200      259,200   256.0x

class FullAttention(nn.Module):
    def __init__(self, d, heads=4):
        super().__init__()
        self.mha = nn.MultiheadAttention(d, heads, batch_first=True)
    def forward(self, x):                 # x: (B, L, d)
        return self.mha(x, x, x)[0]

class PatchAttention(nn.Module):
    def __init__(self, P, d, heads=4):
        super().__init__()
        self.embed = nn.Linear(P, d)      # 패치 -> 토큰 임베딩
        self.mha = nn.MultiheadAttention(d, heads, batch_first=True)
    def forward(self, x, P, S):           # x: (B, L)
        tok = self.embed(patchify(x, P, S))   # (B, n, d)
        return self.mha(tok, tok, tok)[0]

torch.manual_seed(0)
B, d, P, S = 16, 64, 16, 8
full, patch = FullAttention(d), PatchAttention(P, d)

print("measured mean forward time over 20 runs (CPU), B=16, d=64")
print(f"{'L':>5} {'full_ms':>9} {'patch_ms':>9} {'speedup':>8}")
for L in [96, 336, 720]:
    x_full = torch.randn(B, L, d)         # L개 토큰에 대한 완전 어텐션
    x_raw  = torch.randn(B, L)            # 패치 경로는 원시 시계열에서 시작
    for _ in range(3):                    # 워밍업
        full(x_full); patch(x_raw, P, S)
    t0 = time.perf_counter()
    for _ in range(20): full(x_full)
    t1 = time.perf_counter()
    for _ in range(20): patch(x_raw, P, S)
    t2 = time.perf_counter()
    fm, pm = (t1 - t0)/20*1000, (t2 - t1)/20*1000
    print(f"{L:5d} {fm:9.2f} {pm:9.2f} {fm/pm:7.2f}x")
measured mean forward time over 20 runs (CPU), B=16, d=64
    L   full_ms  patch_ms  speedup
   96      3.27      0.83    3.96x
  336     13.62      1.46    9.32x
  720     51.98      2.67   19.45x

이론 표는 평평한 비율을 보여 줍니다. \(S=16\)이면 어텐션 비용은 \(L\)과 무관하게 \(S^2 = 256\)에 가까운 상수 배로 줄어듭니다. 패칭은 성장의 차수를 바꾸지 않기 때문입니다. 여전히 \(O((L/S)^2)\)이며, 상수만 작아질 뿐입니다. 실측 시간은 운영상의 이야기를 들려줍니다. 속도 향상이 \(L\)과 함께 커집니다(이 장비에서 \(L=96\)에서 약 8배였다가 \(L=720\)에서 80배를 넘어섭니다). 짧은 길이에서는 고정 오버헤드가 지배하고, 긴 길이에서는 이차 항이 지배하면서 토큰 감소가 가장 크게 작용하기 때문입니다. 이것이 Informer와의 핵심 차이를 구체적으로 보여 줍니다. PatchTST는 이차 비용의 상수를 낮추고, Informer는 점근 차수 자체를 낮춥니다.

4. 두 경로의 비교

측면 Informer (ProbSparse) PatchTST (패칭)
바꾸는 대상 \(L\times L\) 행렬의 어느 항목을 계산할지 토큰 수 \(L\) 자체
어텐션 비용 \(O(L \log L)\) \(O((L/S)^2)\) — 이차, 작은 상수
근사인가? 예 (키 샘플링, 상위 \(u\) 쿼리) 아니오 (더 적은 토큰에 완전 어텐션)
부가 장치 디스틸링 + 생성형 디코더 채널 독립성 + 인스턴스 정규화
실패 조건 어텐션이 롱테일이 아니고 실제로 밀집일 때 패치가 \(P\)보다 빠른 동역학을 가릴 때
발표 AAAI 2021 (Zhou 기타 2021년) ICLR 2023 (Nie 기타 2023년)

표는 진짜 트레이드오프를 드러냅니다. Informer의 \(O(L\log L)\)은 점근적으로 더 우수하므로 극단적으로 큰 \(L\)에서는 결국 비용에서 이겨야 합니다. 다만 품질이 데이터에 의존하는 근사를 대가로 치릅니다. 이후 연구는 여러 효율적 어텐션 트랜스포머가 표준 LSTF 벤치마크에서 단순한 선형 기준선을 뚜렷이 이기지 못했다고 주장했습니다 (Zeng 기타 2023년). PatchTST는 부분적으로 그에 대한 응답으로 등장했습니다. 정확한 어텐션을 고수했고, 저자들은 패칭에 채널 독립성을 더한 방식이 앞선 효율적 트랜스포머들과 강력한 선형 기준선을 모두 능가했다고 보고했습니다 (Nie 기타 2023년). 이 순위는 벤치마크와 예측 구간에 의존하므로, 확정된 결론이 아니라 증거로 읽어야 합니다.

5. 한계

여기 실험은 CPU에서 작은 텐서로 어텐션 비용만 분리한 것입니다. 메커니즘과 스케일링을 보여 줄 뿐 종단 간 예측 정확도를 보여 주지 않으며, 밀리초 절댓값은 장비에 따라 다릅니다. 여기 제시한 ProbSparse 구현은 선택 단계의 교육용 버전이지, 디스틸링 스택까지 포함한 Informer의 완전한 커널이 아닙니다. PatchTST의 “이차이되 상수가 작은” 비용은, 아주 긴 윈도우에서는 패치 모델도 여전히 비싸질 수 있음을 뜻합니다. 그래서 원시 길이가 아니라 토큰 수를 지켜봐야 합니다. 두 모델 어느 것도 “주어진 시계열에 어텐션이 애초에 올바른 귀납 편향인가”라는 더 깊은 질문을 없애지는 못합니다. 매끄럽고 강하게 선형적인 시계열에서는 한 층짜리 선형 사상이 둘 중 어느 것과도 견줄 수 있으며 (Zeng 기타 2023년), 올바른 선택은 데이터에 실제로 얼마나 비선형적인 시간 교차 구조가 들어 있는지에 달려 있습니다.

6. 결론

Informer와 PatchTST는 같은 \(O(L^2)\) 문제를, 같은 식에서 서로 다른 기호를 고쳐 풉니다. Informer는 행렬을 고칩니다. 신호를 담은 쿼리에 대해서만 어텐션을 계산하고, 데이터에 의존하는 근사를 받아들여 \(O(L\log L)\)에 도달합니다. PatchTST는 시퀀스를 고칩니다. 스텝을 패치로 묶어 정확한 어텐션이 한 자릿수 배수만큼 적은 토큰 위에서 돌게 하며, 이차 형태는 유지하되 상수를 줄이고, 부수적으로 모델에 더 의미 있는 토큰을 줍니다. 어느 쪽이 더 중요한지는 윈도우 길이와, 그 시계열의 어텐션이 실제로 희소한지에 달려 있습니다. Informer의 점근적 이점은 큰 \(L\)에서야 값을 하고, PatchTST의 상수 배 이점은 즉각적이며 공개된 LSTF 벤치마크에서는 더 나은 정확도까지 함께 보고되었습니다. 어느 쪽을 가정하기 전에, 당신의 데이터에서 토큰 수와 어텐션 밀도를 측정하십시오.
